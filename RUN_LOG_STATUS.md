# Run Log 개발 현황

> 기준일: 2026-07-12
> 기준: `RUNFOLIO_ROADMAP.md`의 장기 목표와 현재 저장소 전체 코드 비교
> 제품명: 문서의 과거 명칭 `Runfolio`는 모두 **Run Log**를 의미한다.

## 요약

Run Log는 현재 **iOS 17 이상을 대상으로 하는 SwiftUI·HealthKit 앱**이다. 로드맵이 전제한 Web MVP는 존재하지 않지만, iOS 코드에서는 핵심 MVP인 개별 경로 재생, 상대 페이스 색상, 월간 누적 지도가 이미 상당 부분 구현되어 있다.

현재 가장 큰 과제는 새 화면 추가보다 **핵심 러닝 데이터의 정확성과 회귀 방지 기반을 확보하는 것**이다. 테스트 타깃이 없고, 여러 `HKWorkoutRoute` 조각 중 첫 번째만 사용하며, “현재 페이스”가 실제 순간/구간 페이스가 아닌 시작부터 현재까지 평균이다. 전체 통계라는 UI 문구와 달리 실제 조회 범위도 최근 1년이다.

## 1. 로드맵의 전체 목표 기능

### 핵심 기록 경험

- 전체 러닝 경로, 시작/종료점, 현재 위치 표시
- 시간 순 경로 재생, 일시정지, 리셋, 배속, 타임라인 탐색
- 카메라 자동 추적과 전체 경로 보기
- 경과 시간, 현재/남은 거리, 현재/평균 페이스, 진행률, 구간 표시
- GPS 이상치·정지 구간을 보정한 페이스 기반 경로 색상
- 전체 화면 재생

### 아카이브와 실제 데이터

- 정렬·검색·필터가 가능한 전체 기록 목록과 상세 화면
- 월간 누적 지도, 지도와 목록 선택 연동, 월간 요약
- 연간 통계, 그래프, 히트맵, 개인 최고 기록
- GPX 우선 업로드와 JSON/CSV, 이후 TCX/FIT 및 외부 서비스 연동
- 좌표 검증, 정렬, 중복/GPS 이상치 제거, 거리·페이스·고도 계산
- 계정, 데이터베이스, 기록 CRUD, 공개 범위, 동기화

### 차별화·소셜·게임화

- 과거 기록과 동일 경로 비교, 개인 최고 기록
- 시간 또는 거리 기준 Ghost Runner/Ghost Race
- 친구, 피드, 랭킹, 응원, 댓글, 프로필
- 목표, 연속 러닝, 배지, 챌린지, 레벨, Route Collection, Weekly League

### 제품화와 플랫폼

- 개인정보·위치정보 보호, 집/회사 주변 경로 숨김
- 오류 로깅, 성능·분석 이벤트, 접근성, 자동화 테스트, CI/CD
- Web 모바일 반응형과 PWA
- Android 패키징/네이티브 앱, Health Connect, 알림, 웨어러블
- 장기적으로 자체 GPS 측정, 백그라운드 기록, 음성 안내, 오프라인 동기화

## 2. 현재 코드에서 구현된 기능

### 앱·데이터 입력

- SwiftUI 앱 진입점과 탭 기반 요약/월별/이력 화면 (`RunHealthPrototypeApp.swift`, `ContentView.swift`)
- HealthKit workout, workout route, 심박수 읽기 권한 요청
- 최근 365일 running workout 조회와 평균 심박수 병렬 조회 (`HealthKitService.swift`)
- workout 요약의 JSON 디스크 캐시와 최신순 정렬 (`RunWorkoutCacheStore.swift`)
- 앱 세션 동안 route 및 route 없음 결과 메모리 캐시

### 요약·목록·상세

- 최근 러닝 카드와 Apple Map 경로 미리보기
- 전체/이번 달/올해/최근 7일/최근 30일 횟수·거리 요약
- 최신순 러닝 이력과 기간 카드에서 필터된 목록으로 이동
- 날짜, 거리, 시간, 평균 페이스, 평균 심박수 상세 표시

### 지도와 리플레이

- Apple MapKit 및 Google Maps SDK 10.0.0 선택 구조
- 정적 전체 경로 polyline, 시작/종료 마커, 자동 영역 맞춤
- timestamp 기반 재생/일시정지/리셋
- 포인트 단위 타임라인 슬라이더와 10x/20x/40x 배속
- 현재 위치 마커와 재생 경로 점진 표시
- Apple Map 표준 추적 및 진행 방향 기반 cinematic 3D 카메라
- 사용자가 정지 상태에서 지도를 움직이면 표준 카메라 추적 해제
- 전체 화면 리플레이와 오버레이 숨김
- 누적 거리, 경과 시간, 현재까지 평균 페이스 표시
- 러닝 내부 33/67 백분위 기준 빠름/보통/느림 3색 경로
- 최대 900포인트 다운샘플링과 보간된 현재 위치 이동

### 월간 아카이브와 UI

- 최근 12개월 선택
- 월간 횟수, 거리, 시간, 평균 페이스
- 월별 workout route 병렬 조회와 누적 지도
- route 없음/실패/표시 건수 상태 표시
- 월간 기록 목록에서 상세 화면 이동
- 공통 카드·배지·색상·지도 테마 디자인 시스템
- 표시명과 화면 브랜드는 `RunLog`로 설정

## 3. 부분 구현 또는 미완성 기능

### 데이터 정확성

- 한 workout에 `HKWorkoutRoute`가 여러 개면 `routes.first`만 읽는다. 일시정지/재개 등으로 route가 나뉜 기록은 경로 일부가 누락될 수 있다.
- route 포인트의 시간 정렬, 중복 제거, 좌표 유효성 검사, GPS 점프/정지 구간 보정이 없다.
- 포인트 수 기준 단순 다운샘플링은 시간·거리 분포를 보존하지 않아 경로와 페이스 색상을 왜곡할 수 있다.
- “현재 페이스”는 현재 구간/이동 창 페이스가 아니라 시작부터 현재까지의 평균 페이스다.
- 비현실적인 페이스는 임계값 계산에서만 제외되고 지도에서는 `normal`로 표시된다.
- 고도는 route point에 저장하지만 합산·표시하지 않는다. cadence, power, 누적 원본 거리도 없다.

### 리플레이

- 로드맵의 1x/2x/4x 대신 10x/20x/40x만 제공한다.
- 남은 거리, 진행률 숫자, 현재 구간 번호는 표시하지 않는다.
- Apple Map만 페이스별 polyline과 카메라 추적을 지원한다. Google Map 리플레이는 단색 경로와 전체 영역 고정이며 현재 위치만 이동한다.
- 두 개의 항상 연결된 타이머가 재생 중이 아닐 때도 이벤트를 발생시켜 불필요한 에너지 사용 가능성이 있다.
- `FullscreenReplayView.swift` 한 파일에 화면, 상태 모델, 카메라 수학, 데이터 가공이 1,200줄 이상 결합되어 있다.

### 아카이브·기록 관리

- “전체/누적” 통계와 목록의 실제 원본은 최근 365일 조회 결과라 진정한 전체 기록이 아니다.
- 월간 화면은 최근 12개월만 선택 가능하며 이전/다음 월 버튼은 없다.
- 월별 경로가 모두 같은 색이고, 지도 경로 선택·강조 및 목록과 지도 동기화가 없다.
- 목록은 최신순만 제공한다. 정렬 변경, 검색, 기간/거리 필터, 경로 미리보기, 제목, 고도, 유형, 공개 범위가 없다.
- 로컬 캐시는 workout 요약만 저장한다. route 영속 캐시, 증분 동기화, 수정/삭제는 없다.
- 캐시가 없는 첫 실행에서는 자동 HealthKit 갱신을 시도하지 않고 사용자가 설정에서 권한과 조회를 각각 실행해야 한다.

### 제품화

- Google Maps API 키는 빌드 설정 주입 구조만 있으며 저장소에 개발/배포 환경 구성 안내가 없다.
- 접근성 식별자, 오류 로깅, 분석, 개인정보/위치 숨김 정책이 없다.
- README는 초기 프로토타입 수준 설명이라 현재 탭, 캐시, 월간 지도, 리플레이, Google Map 기능을 반영하지 않는다.
- 화면 표시명은 RunLog지만 Xcode 프로젝트/타깃/번들 ID와 소스 폴더는 `RunHealthPrototype` 명칭을 유지한다.

- 서버, 데이터베이스, 인증, 사용자별 저장, 기록 CRUD, 공개 범위
- 과거 기록 비교, 동일 경로 판별, 개인 최고 기록, Ghost Runner
- 친구, 피드, 랭킹, 반응, 댓글, 프로필
- 목표, 배지, 연속 러닝, 챌린지, 레벨, 경로 수집, 리그
- 연간 그래프·지도·히트맵과 자주 달린 지역/경로 분석

- Web 앱, 모바일 Web, PWA
- Android 앱, Capacitor/React Native/Kotlin, Health Connect, Wear OS
- XCTest, UI 테스트, lint, CI/CD, 운영 모니터링

## 5. 로드맵 문서와 실제 코드의 차이

| 항목 | 로드맵의 기존 전제 | 실제 저장소 |
|---|---|---|
| 제품명 | Runfolio | 사용자 노출명은 RunLog, 현재 정식 명칭은 Run Log |
| 주 플랫폼 | Web MVP 후 PWA/Android | Web/Android 코드 없이 iOS SwiftUI 앱이 중심 |
| 데이터 입력 | GPX 업로드를 MVP 1순위로 제안 | Apple HealthKit이 유일한 실제 데이터 원본 |
| 지도 | 라이브러리 미확정, 정적 지도만 확인 | Apple MapKit + 선택적 Google Maps SDK |
| 리플레이 | 남은 주요 개발 범위 | 재생·슬라이더·배속·마커·카메라·전체 화면까지 구현 |
| 페이스 시각화 | 미구현 목표 | 3단계 상대 페이스 색상 구현, 정확도 보정은 미완성 |
| 월간 아카이브 | 미구현 목표 | 최근 12개월 요약·누적 지도·목록 구현, 선택 연동은 미완성 |
| 백엔드/계정 | 향후 필요 | 없음 |
| Android | 단계별 전략만 정의 | 관련 코드/설정 없음 |

따라서 로드맵은 장기 제품 목표로는 유효하지만, 개발 순서는 “Web MVP 착수”보다 현재 iOS 핵심 데이터의 정확성·테스트·완성도를 먼저 높이는 방향으로 조정해야 한다.

## 6. 주요 오류와 기술 부채

### P0 — 신뢰성과 검증

1. **분할 route 누락 가능성**: `HealthKitService.fetchUncachedRoute`가 첫 번째 `HKWorkoutRoute`만 처리한다.
2. **자동화 테스트 전무**: 순수 계산 함수가 다수 있지만 XCTest 타깃이 없어 거리, 시간, 다운샘플링, 페이스 분류, 월/기간 경계를 보호하지 못한다.
3. **실제 빌드 검증 환경 불일치**: Xcode 26.6과 설치된 CoreSimulator/플랫폼 구성의 빌드 버전이 맞지 않아 simulator와 generic iOS destination을 사용할 수 없다. Swift parser 검사는 통과했지만 타입체크·링크·실기기 실행 성공으로 간주할 수 없다.

### P1 — 데이터 의미와 사용자 신뢰

1. “전체 기록”과 “누적 거리”가 최근 1년 데이터만 집계한다.
2. 현재 페이스 라벨이 실제 계산 의미와 다르다.
3. GPS 이상치·중복·시간 역전·비현실적 속도를 정규화하지 않는다.
4. HealthKit 읽기 권한은 허용/거부 상태를 직접 알려주지 않는데, 앱 상태는 요청 완료만 표시해 실제 데이터 없음과 권한 거부를 구분하지 못한다.
5. route 없음 결과가 앱 세션 동안 고정 캐시되어 Health 데이터가 나중에 보완돼도 재시도되지 않는다.

### P2 — 구조·성능·운영

1. 리플레이 핵심 파일이 지나치게 커 UI와 도메인 로직 테스트가 어렵다.
2. 재생/시각 타이머가 일시정지 중에도 연결돼 있다.
3. Google Map은 Apple Map과 기능 동등성이 없지만 같은 제공자 선택으로 노출된다.
4. `RunWorkoutCacheStore`는 `@unchecked Sendable`로 동시성 안전을 컴파일러 대신 개발자가 보장한다.
5. `.DerivedData/`가 `.gitignore`의 `DerivedData/` 패턴에 잡히지 않아 대량의 생성물이 untracked 상태로 노출된다.
6. 내부 프로젝트명·번들 ID가 프로토타입 이름이고 README와 실제 기능이 어긋난다.

## 7. Web, 모바일, Android 현재 구조

### Web

구현 없음. `package.json`, Web 프레임워크, PWA manifest, service worker, Web 지도 코드가 없다. 로드맵의 Web/PWA 내용은 향후 전략이다.

### 모바일/iOS

- 단일 iPhone 앱 타깃, iOS 17.0+, Swift 5 설정
- SwiftUI `NavigationStack` + `TabView`, 별도 전역 상태관리 라이브러리 없음
- 각 화면의 `@State`, `@StateObject`, `@AppStorage`와 서비스 객체로 상태 관리
- HealthKit이 workout/route/heart rate 데이터 원본
- MapKit 기본, Google Maps SDK 10.0.0 선택 지원
- workout 요약은 Application Support의 JSON 캐시, route는 메모리 캐시
- 서버/API/데이터베이스 없음

### Android

구현 없음. Gradle, Android Manifest, Kotlin/Java, Capacitor, React Native, Health Connect 설정이 없다. iOS SwiftUI 코드는 Android에서 직접 재사용할 수 없으며, 공통 데이터 모델/API가 아직 없어 플랫폼 확장 전에 데이터 계약 설계가 필요하다.

## 8. 남은 작업 권장 우선순위

1. **P0: 빌드 환경 복구와 테스트 기반** — Xcode 플랫폼/CoreSimulator 정합성을 맞추고 XCTest 타깃을 추가해 핵심 계산과 route 병합을 검증한다.
2. **P0: HealthKit route 정규화** — 모든 route 조각 병합, 시간 정렬, 중복·유효하지 않은 좌표·시간 역전 처리.
3. **P1: 지표 의미 교정** — 현재 페이스를 이동 창/구간 페이스로 계산하거나 라벨을 “현재까지 평균”으로 변경하고, 전체 기록의 365일 범위를 UI와 일치시킨다.
4. **P1: 리플레이 안정화** — 재생 중에만 타이머 활성화, 긴/정지 포함 경로 검증, Apple/Google 기능 차이 명시.
5. **P1: 월간 아카이브 상호작용** — route별 색상·선택·강조, 지도/목록 동기화, 더 긴 월 범위.
6. **P2: 기록 탐색** — 정렬, 검색, 기간/거리 필터, 고도와 경로 미리보기.
7. **P2: 로컬 데이터 완성** — route 영속 캐시, 증분 갱신, 실패/권한 UX, README와 구성 문서 갱신.
8. **P3: 차별화** — 과거 기록 비교, 동일 경로 판별, Ghost Runner, 연간 포트폴리오.
9. **P4: 서비스화** — 개인정보 정책과 공개 범위를 먼저 설계한 뒤 계정/백엔드, 소셜, 게임화.
10. **P5: 추가 플랫폼** — 제품 요구가 확인되면 Web/PWA와 Android의 공통 API·데이터 계약부터 설계.

## 9. 바로 이어서 진행할 다음 작업

### 선정 작업

**HealthKit의 분할 route를 빠짐없이 병합하고, route 정규화 로직에 대한 단위 테스트를 추가한다.**

이 작업은 지도, 리플레이, 월간 누적 지도 모두가 공유하는 입력 데이터의 누락 가능성을 제거한다. 화면 디자인과 기존 탐색 구조는 바꾸지 않는다.

### 수정 전에 보고할 변경 파일

- `RunHealthPrototype/HealthKitService.swift`: 모든 `HKWorkoutRoute`의 location을 읽고 오류·취소를 올바르게 전달하도록 조회 흐름 수정
- `RunHealthPrototype/RunRoute.swift`: 필요하면 순수 route point 정규화 함수 또는 모델 보조 로직 추가
- `RunHealthPrototypeTests/RunRouteProcessingTests.swift` (신규): 분할 route 병합, 시간 정렬, 중복/빈 route, 거리·경과 시간 회귀 테스트
- `RunHealthPrototype.xcodeproj/project.pbxproj`: XCTest 타깃과 테스트 파일 등록

### 구현 계획

1. 각 `HKWorkoutRoute`의 모든 location batch를 수집한다.
2. 여러 route 결과를 하나로 합치고 timestamp 기준으로 안정 정렬한다.
3. 완전히 동일한 timestamp·좌표 중복과 유효하지 않은 좌표를 작은 순수 함수에서 제거한다.
4. 정상 단일 route 동작은 그대로 유지하고, 결과가 비면 기존 `routeUnavailable` 흐름을 사용한다.
5. 순수 정규화 함수와 리플레이 거리/시간 계산에 XCTest를 추가한다.
6. Swift parser, XCTest, iOS 빌드를 실행하고 Xcode 환경 문제와 코드 실패를 분리해 보고한다.

이 문서 작성 단계에서는 위 소스 코드를 수정하지 않았다. 다음 구현은 이 계획을 확인한 뒤 진행한다.
