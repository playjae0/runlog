# Run Log 개발 현황

> 기준일: 2026-07-12
> 기준: `RUNFOLIO_ROADMAP.md`의 장기 목표와 현재 저장소 전체 코드 비교
> 제품명: 문서의 과거 명칭 `Runfolio`는 모두 **Run Log**를 의미한다.

## 요약

Run Log는 현재 **iOS 17 이상을 대상으로 하는 SwiftUI·HealthKit·Apple MapKit 앱**이다. 로드맵이 전제한 Web MVP는 존재하지 않지만, iOS 코드에서는 핵심 MVP인 개별 경로 재생, 상대 페이스 색상, 월간 누적 지도가 이미 상당 부분 구현되어 있다.

### Sprint 진행

- Phase A 완료: Google Maps SDK, API 키, 제공자 선택 UI를 제거하고 모든 지도 화면을 Apple MapKit으로 단일화했다.
- Phase B 완료: `.DerivedData/` 등 생성물을 제외하고 XCTest 타깃과 거리·시간·다운샘플링·월 경계 회귀 테스트를 추가했다.
- Phase C 완료: 한 workout의 모든 `HKWorkoutRoute`를 병합하고 timestamp 안정 정렬, 유효 좌표 필터, 완전 동일 중복 제거를 적용했다.
- Phase D 완료: 페이스와 통계 라벨을 실제 의미에 맞추고, 리플레이를 실제 경과 시각 기반·재생 중 전용 타이머로 변경했으며 남은 거리와 진행률을 추가했다.
- Phase E 완료: 월 범위를 24개월로 확장하고 이전/다음 이동, 미래 월 방지, 목록 선택에 따른 경로 강조·줌, 상세 이동 및 이력 정렬을 추가했다.
- Phase F 완료: workout 캐시 schema version, route 없음 캐시 10분 만료, 권한 요청 후 자동 조회, 저장소 위생과 README를 실제 코드에 맞게 정리했다.

Sprint A~F의 안전한 코드 작업은 완료했다. 현재 가장 큰 차단 요인은 **Xcode 26.6과 설치된 CoreSimulator/iOS 플랫폼 구성 불일치**이며, 추가된 XCTest 실행과 실기기 HealthKit·지도 검증이 필요하다.

## 1. 로드맵의 전체 목표 기능

### 핵심 기록 경험

- 전체 러닝 경로, 시작/종료점, 현재 위치 표시
- 시간 순 경로 재생, 일시정지, 리셋, 배속, 타임라인 탐색
- 카메라 자동 추적과 전체 경로 보기
- 경과 시간, 현재/남은 거리, 현재/평균 페이스, 진행률, 구간 표시
- GPS 이상치·정지 구간을 보정한 페이스 기반 경로 색상
- 전체 화면 재생

### 아카이브와 실제 데이터

- 정렬·검색·필터가 가능한 전체 기록 목록과 상세 화면
- 월간 누적 지도, 지도와 목록 선택 연동, 월간 요약
- 연간 통계, 그래프, 히트맵, 개인 최고 기록
- GPX 우선 업로드와 JSON/CSV, 이후 TCX/FIT 및 외부 서비스 연동
- 좌표 검증, 정렬, 중복/GPS 이상치 제거, 거리·페이스·고도 계산
- 계정, 데이터베이스, 기록 CRUD, 공개 범위, 동기화

### 차별화·소셜·게임화

- 과거 기록과 동일 경로 비교, 개인 최고 기록
- 시간 또는 거리 기준 Ghost Runner/Ghost Race
- 친구, 피드, 랭킹, 응원, 댓글, 프로필
- 목표, 연속 러닝, 배지, 챌린지, 레벨, Route Collection, Weekly League

### 제품화와 플랫폼

- 개인정보·위치정보 보호, 집/회사 주변 경로 숨김
- 오류 로깅, 성능·분석 이벤트, 접근성, 자동화 테스트, CI/CD
- Web 모바일 반응형과 PWA
- Android 패키징/네이티브 앱, Health Connect, 알림, 웨어러블
- 장기적으로 자체 GPS 측정, 백그라운드 기록, 음성 안내, 오프라인 동기화

## 2. 현재 코드에서 구현된 기능

### 앱·데이터 입력

- SwiftUI 앱 진입점과 탭 기반 요약/월별/이력 화면 (`RunHealthPrototypeApp.swift`, `ContentView.swift`)
- HealthKit workout, workout route, 심박수 읽기 권한 요청
- 최근 365일 running workout 조회와 평균 심박수 병렬 조회 (`HealthKitService.swift`)
- workout 요약의 JSON 디스크 캐시와 최신순 정렬 (`RunWorkoutCacheStore.swift`)
- 앱 세션 동안 route 및 route 없음 결과 메모리 캐시

### 요약·목록·상세

- 최근 러닝 카드와 Apple Map 경로 미리보기
- 최근 1년/이번 달/올해/최근 7일/최근 30일 횟수·거리 요약
- 최신순 러닝 이력과 기간 카드에서 필터된 목록으로 이동
- 날짜, 거리, 시간, 평균 페이스, 평균 심박수 상세 표시

### 지도와 리플레이

- Apple MapKit 단일 지도 구조
- 정적 전체 경로 polyline, 시작/종료 마커, 자동 영역 맞춤
- timestamp 기반 재생/일시정지/리셋
- 포인트 단위 타임라인 슬라이더와 10x/20x/40x 배속
- 현재 위치 마커와 재생 경로 점진 표시
- Apple Map 표준 추적 및 진행 방향 기반 cinematic 3D 카메라
- 사용자가 정지 상태에서 지도를 움직이면 표준 카메라 추적 해제
- 전체 화면 리플레이와 오버레이 숨김
- 누적 거리, 경과 시간, 현재까지 평균 페이스 표시
- 러닝 내부 33/67 백분위 기준 빠름/보통/느림 3색 경로
- 최대 900포인트 다운샘플링과 보간된 현재 위치 이동

### 월간 아카이브와 UI

- 최근 24개월 선택과 이전/다음 월 이동
- 월간 횟수, 거리, 시간, 평균 페이스
- 월별 workout route 병렬 조회와 누적 지도
- route 없음/실패/표시 건수 상태 표시
- 월간 기록 목록에서 상세 화면 이동
- 공통 카드·배지·색상·지도 테마 디자인 시스템
- 표시명과 화면 브랜드는 `RunLog`로 설정

## 3. 부분 구현 또는 미완성 기능

### 데이터 정확성

- route 포인트 병합·정렬·명백한 중복과 유효하지 않은 좌표 제거는 구현됐다. GPS 점프/정지 구간 보정은 원본 훼손 위험 때문에 아직 적용하지 않았다.
- 포인트 수 기준 단순 다운샘플링은 시간·거리 분포를 보존하지 않아 경로와 페이스 색상을 왜곡할 수 있다.
- 이동 창 페이스는 아직 없으며 UI는 계산 의미대로 “현재까지 평균”으로 표시한다.
- 비현실적인 페이스는 임계값 계산에서만 제외되고 지도에서는 `normal`로 표시된다.
- 고도는 route point에 저장하지만 합산·표시하지 않는다. cadence, power, 누적 원본 거리도 없다.

### 리플레이

- 로드맵의 1x/2x/4x 대신 10x/20x/40x만 제공한다.
- 남은 거리와 진행률은 표시하며 현재 구간 번호는 표시하지 않는다.
- 페이스별 polyline과 카메라 추적은 Apple MapKit으로 제공한다.
- 리플레이 타이머는 재생 중에만 생성되고 일시정지·화면 종료·백그라운드 진입 시 해제된다.
- `FullscreenReplayView.swift` 한 파일에 화면, 상태 모델, 카메라 수학, 데이터 가공이 1,200줄 이상 결합되어 있다.

### 아카이브·기록 관리

- 통계와 목록의 실제 원본은 최근 365일이며 관련 요약 UI는 “최근 1년”으로 표시한다.
- 월간 화면은 최근 24개월을 제공하며 미래 월 이동을 막는다. 24개월 이전 기록은 아직 선택할 수 없다.
- 월별 경로는 러닝별 색상을 사용하고 목록 선택 시 강조·줌한다. 지도 선 직접 선택은 실기기 hit-testing 검증이 필요해 미구현이다.
- 목록은 최신·오래된·거리·페이스 정렬을 제공한다. 검색, 기간/거리 필터, 경로 미리보기, 제목, 고도, 유형, 공개 범위는 없다.
- 로컬 캐시는 workout 요약만 저장한다. route 영속 캐시, 증분 동기화, 수정/삭제는 없다.
- 첫 권한 요청 절차가 완료되면 자동 조회하고 이후 설정에서 수동 갱신할 수 있다.

### 제품화

- 접근성 식별자, 오류 로깅, 분석, 개인정보/위치 숨김 정책이 없다.
- route 영속 캐시는 민감한 위치정보와 무효화 정책 때문에 이번 Sprint에서 도입하지 않았다.
- 화면 표시명은 RunLog지만 Xcode 프로젝트/타깃/번들 ID와 소스 폴더는 `RunHealthPrototype` 명칭을 유지한다.

- 과거 기록 비교, 동일 경로 판별, 개인 최고 기록, Ghost Runner
- 친구, 피드, 랭킹, 반응, 댓글, 프로필
- 목표, 배지, 연속 러닝, 챌린지, 레벨, 경로 수집, 리그
- 연간 그래프·지도·히트맵과 자주 달린 지역/경로 분석
- 자체 GPS 측정, 백그라운드 추적, 음성 안내, 오프라인 동기화

- Android 앱, Capacitor/React Native/Kotlin, Health Connect, Wear OS
- UI 테스트, lint, CI/CD, 운영 모니터링

## 5. 로드맵 문서와 실제 코드의 차이

| 항목 | 로드맵의 기존 전제 | 실제 저장소 |
|---|---|---|
| 제품명 | Runfolio | 사용자 노출명은 RunLog, 현재 정식 명칭은 Run Log |
| 주 플랫폼 | Web MVP 후 PWA/Android | Web/Android 코드 없이 iOS SwiftUI 앱이 중심 |
| 데이터 입력 | GPX 업로드를 MVP 1순위로 제안 | Apple HealthKit이 유일한 실제 데이터 원본 |
| 지도 | 라이브러리 미확정, 정적 지도만 확인 | Apple MapKit 단일 구현 |
| 리플레이 | 남은 주요 개발 범위 | 재생·슬라이더·배속·마커·카메라·전체 화면까지 구현 |
| 페이스 시각화 | 미구현 목표 | 3단계 상대 페이스 색상 구현, 정확도 보정은 미완성 |
| 월간 아카이브 | 미구현 목표 | 최근 24개월 요약·누적 지도·목록 선택 강조 구현, 지도 선 직접 선택은 미완성 |
| 백엔드/계정 | 향후 필요 | 없음 |
| Android | 단계별 전략만 정의 | 관련 코드/설정 없음 |

따라서 로드맵은 장기 제품 목표로는 유효하지만, 개발 순서는 “Web MVP 착수”보다 현재 iOS 핵심 데이터의 정확성·테스트·완성도를 먼저 높이는 방향으로 조정해야 한다.

## 6. 주요 오류와 기술 부채

### P0 — 신뢰성과 검증

1. **테스트 실행 환경 차단**: XCTest 타깃과 핵심 순수 계산·route 정규화 테스트는 추가했지만 현재 CoreSimulator 버전 불일치로 실행하지 못했다.
2. **실제 빌드 검증 환경 불일치**: Xcode 26.6과 설치된 CoreSimulator/플랫폼 구성의 빌드 버전이 맞지 않아 simulator와 generic iOS destination을 사용할 수 없다. Swift parser 검사는 통과했지만 타입체크·링크·실기기 실행 성공으로 간주할 수 없다.

### P1 — 데이터 의미와 사용자 신뢰

1. GPS 점프·정지 구간과 비현실적 속도는 아직 보정하지 않는다.
2. HealthKit 읽기 권한은 허용/거부 상태를 직접 알려주지 않는데, 앱 상태는 요청 완료만 표시해 실제 데이터 없음과 권한 거부를 구분하지 못한다.
3. route 없음 결과는 10분 뒤 재시도되지만 route 자체는 디스크에 영속 저장하지 않는다.

### P2 — 구조·성능·운영

1. 리플레이 핵심 파일이 지나치게 커 UI와 도메인 로직 테스트가 어렵다.
2. `RunWorkoutCacheStore`는 `@unchecked Sendable`로 동시성 안전을 컴파일러 대신 개발자가 보장한다.
3. 내부 프로젝트명·번들 ID가 프로토타입 이름을 유지한다.

## 7. Web, 모바일, Android 현재 구조

### Web

구현 없음. `package.json`, Web 프레임워크, PWA manifest, service worker, Web 지도 코드가 없다. 로드맵의 Web/PWA 내용은 향후 전략이다.

### 모바일/iOS

- 단일 iPhone 앱 타깃, iOS 17.0+, Swift 5 설정
- SwiftUI `NavigationStack` + `TabView`, 별도 전역 상태관리 라이브러리 없음
- 각 화면의 `@State`, `@StateObject`, `@AppStorage`와 서비스 객체로 상태 관리
- HealthKit이 workout/route/heart rate 데이터 원본
- Apple MapKit 단일 지도 구현
- workout 요약은 Application Support의 JSON 캐시, route는 메모리 캐시
- 서버/API/데이터베이스 없음

### Android

구현 없음. Gradle, Android Manifest, Kotlin/Java, Capacitor, React Native, Health Connect 설정이 없다. iOS SwiftUI 코드는 Android에서 직접 재사용할 수 없으며, 공통 데이터 모델/API가 아직 없어 플랫폼 확장 전에 데이터 계약 설계가 필요하다.

## 8. 남은 작업 권장 우선순위

1. **P0: 빌드 환경 복구** — Xcode 플랫폼/CoreSimulator 정합성을 맞추고 추가된 XCTest를 실제 실행한다.
2. **P1: 리플레이 실제 기기 검증** — 긴 경로, 일시정지, 백그라운드 전환, cinematic 카메라를 실기기에서 확인한다.
3. **P1: 월간 지도 직접 선택** — MapKit overlay hit-testing을 실기기에서 검증한 뒤 지도→목록 동기화를 추가한다.
4. **P2: 기록 탐색** — 검색, 기간/거리 필터, 고도와 경로 미리보기.
5. **P2: 로컬 데이터 완성** — 개인정보·무효화 정책을 정한 뒤 route 영속 캐시와 증분 갱신을 검토한다.
6. **P3: 차별화** — 과거 기록 비교, 동일 경로 판별, Ghost Runner, 연간 포트폴리오.
7. **P4: 서비스화** — 개인정보 정책과 공개 범위를 먼저 설계한 뒤 계정/백엔드, 소셜, 게임화.
8. **P5: 추가 플랫폼** — 제품 요구가 확인되면 Web/PWA와 Android의 공통 API·데이터 계약부터 설계.
