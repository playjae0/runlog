# 1. 프로젝트 개요

Run Log는 러닝 기록을 단순히 숫자와 목록으로 저장하는 앱이 아니다.

핵심 콘셉트는 다음과 같다.

> 사용자의 러닝 기록을 지도 위에 시각적으로 축적하고, 과거의 나 또는 친구의 기록과 비교하며, 게임적인 동기를 통해 다음 러닝을 유도한다.

현재 제품명은 **Run Log**다. 초기 기획의 `Running + Portfolio` 콘셉트는 제품명이 아니라, 사용자의 러닝 기록을 일회성 운동 데이터가 아닌 계속 축적되는 개인 자산처럼 보여주는 제품 방향으로 유지한다.

## 핵심 키워드

- 러닝 기록
- 지도 기반 시각화
- 경로 재생
- 구간별 페이스 분석
- 월간 러닝 아카이브
- 개인 기록 비교
- Ghost Runner
- 친구 기록 비교
- 랭킹
- 챌린지
- 게임화
- 러닝 포트폴리오

# 2. 제품이 제공해야 하는 핵심 경험

사용자가 하나의 러닝 기록을 열었을 때 다음 경험을 제공하는 것이 1차 목표다.

1. 사용자가 달린 전체 경로를 지도에서 확인한다.
2. 재생 버튼을 누르면 실제로 달린 순서에 따라 경로가 그려진다.
3. 현재 위치 마커가 경로를 따라 이동한다.
4. 재생 중 현재 시간, 거리, 페이스가 함께 갱신된다.
5. 빠르게 달린 구간과 느리게 달린 구간이 색상으로 구분된다.
6. 사용자가 원하는 시점으로 타임라인을 이동할 수 있다.
7. 과거의 동일 경로 또는 개인 최고 기록과 비교할 수 있다.
8. 한 달 동안 달린 모든 경로를 하나의 지도에서 확인할 수 있다.
9. 친구와 거리, 속도, 횟수, 특정 구간 기록을 비교할 수 있다.
10. 목표, 배지, 챌린지 등을 통해 다음 러닝 행동이 유도된다.

# 5. 추천 제품 전략

## 5.1 1차 제품 정의

초기에는 Run Log를 다음과 같이 정의하는 것을 권장한다.

> 기존에 보유한 러닝 데이터를 불러와 지도에서 재생하고, 월간 기록과 과거 기록을 비교할 수 있는 러닝 포트폴리오 서비스

초기 버전에서는 스마트폰 GPS로 직접 러닝을 측정하는 기능을 필수로 넣지 않는다.

### 이유

- 백그라운드 GPS 기록은 앱 개발 난도가 높다.
- Android 권한, 배터리, 위치정보 처리 이슈가 크다.
- iOS까지 고려하면 개발 범위가 급격히 커진다.
- Strava, Garmin, Samsung Health 등 기존 앱과 직접 경쟁해야 한다.
- Run Log의 핵심 차별점은 측정 자체보다 시각화, 비교, 축적, 게임화에 있다.

따라서 먼저 **데이터를 보여주는 경험**을 완성한 뒤, 이후 자체 러닝 기록 기능을 추가하는 것이 좋다.

# 7. Phase 1 — 개별 러닝 상세 화면 완성

## 목표

한 번의 러닝 기록을 보는 경험을 완성한다.

---

## 7.1 경로 재생 애니메이션

### 핵심 기능

- 재생
- 일시정지
- 리셋
- 진행률 슬라이더
- 1배속
- 2배속
- 4배속
- 경로 점진적 표시
- 현재 위치 마커 이동
- 카메라 자동 추적
- 슬라이더와 재생 위치 동기화

### 권장 핵심 상태

```ts
type PlaybackSpeed = 1 | 2 | 4;

interface PlaybackState {
  currentIndex: number;
  isPlaying: boolean;
  playbackSpeed: PlaybackSpeed;
  followCamera: boolean;
}
```

재생 상태의 단일 기준은 `currentIndex` 또는 같은 역할을 하는 하나의 값으로 통합한다.

현재 표시되는 경로는 다음과 같이 계산할 수 있다.

```ts
const visibleCoordinates = coordinates.slice(0, currentIndex + 1);
```

현재 위치는 다음과 같이 계산한다.

```ts
const currentCoordinate = coordinates[currentIndex];
```

### 동작 원칙

- 재생하면 `currentIndex`가 증가한다.
- 일시정지하면 현재 위치에서 멈춘다.
- 리셋하면 시작점으로 돌아간다.
- 슬라이더를 이동하면 즉시 해당 위치로 이동한다.
- 슬라이더 이동 후 재생을 이어갈 수 있다.
- 마지막 좌표에 도달하면 자동 종료한다.
- 배속 변경 시 상태가 꼬이지 않아야 한다.
- 컴포넌트가 사라질 때 timer와 animation frame을 정리한다.

---

## 7.2 카메라 추적

### 기능

- 현재 위치를 카메라가 따라간다.
- 사용자가 직접 지도를 이동하면 자동 추적을 일시적으로 해제한다.
- `Follow` 버튼을 통해 추적을 다시 활성화한다.
- 사용자가 원하면 전체 경로 보기로 돌아갈 수 있다.

### 고려사항

- 지도 이동이 지나치게 자주 발생하지 않도록 한다.
- 매 GPS 포인트마다 카메라를 갱신하기보다 간격을 둔다.
- 모바일 화면에서 지도가 흔들리지 않도록 한다.

## 7.3 러닝 상태 정보

재생 중 다음 항목을 표시한다.

interface RunPlaybackMetrics {

## 7.4 페이스 기반 경로 색상

경로 전체를 하나의 색으로 표시하지 않고, 각 구간의 페이스 또는 속도에 따라 색상을 다르게 표시한다.

### 구간 데이터 구조 예시

```ts
interface RunSegment {
  startIndex: number;
  endIndex: number;
  coordinates: Coordinate[];
  distanceMeters: number;
  durationSeconds: number;
  paceSecondsPerKm: number;
  speedMetersPerSecond: number;
}
```

### 페이스 계산

```ts
const paceSecondsPerKm =
  durationSeconds / (distanceMeters / 1000);
```

### 데이터 보정

다음 구간은 제외하거나 보정한다.

- 거리가 거의 없는 구간
- timestamp가 같은 구간
- GPS 점프가 발생한 구간
- 비현실적인 속도가 계산되는 구간
- 장시간 정지 구간

### 초기 색상 분류

첫 버전에서는 한 러닝 내 상대적인 속도를 사용한다.

- 매우 빠름
- 빠름
- 평균
- 느림
- 매우 느림

향후 다음 옵션을 추가할 수 있다.

# 14. Android 개발 전략

Android는 처음부터 모든 기능을 네이티브로 다시 개발하기보다 단계적으로 접근한다.

---

## 14.1 선택지 A — 모바일 Web + PWA

### 설명

향후 Web 앱을 구축할 경우 모바일 반응형으로 만들고 PWA로 설치 가능하게 한다.

### 장점

- 기존 코드 재사용
- 가장 빠른 개발
- 하나의 코드베이스
- 앱처럼 홈 화면 설치 가능
- 지도, 기록 조회, 비교, 업로드에 적합

### 단점

- 백그라운드 GPS 제한
- 센서와 네이티브 기능 제약
- Play Store 앱 경험과 차이
- 플랫폼별 기능 지원 차이

### 적합한 범위

- 러닝 기록 확인
- 경로 재생
- 월간 지도
- 기록 비교
- GPX 업로드
- 피드
- 랭킹
- 챌린지
- 프로필

### 추천

**1차 MVP는 PWA까지 완성하는 것을 권장한다.**

---

## 14.2 선택지 B — WebView 기반 Android 앱

가능한 기술:

- Capacitor
- Ionic
- Android WebView wrapper

### 장점

- Web 코드를 대부분 재사용
- Play Store 배포 가능
- 푸시 알림 등 일부 네이티브 기능 사용 가능
- 초기 앱 전환이 빠름

### 단점

- 복잡한 지도와 애니메이션 성능을 확인해야 함
- 자체 GPS 기록에는 추가 네이티브 작업 필요
- 앱 품질이 WebView처럼 느껴질 수 있음

### 추천 용도

Web MVP가 안정화된 뒤, **조회 및 비교 중심 앱**으로 빠르게 출시할 때 적합하다.

---

## 14.3 선택지 C — React Native

### 장점

- JavaScript 또는 TypeScript 역량 활용 가능
- Android와 iOS 동시 대응
- 네이티브 API 접근 가능
- 지도와 위치 추적 구현 가능
- Web 로직 일부 공유 가능

### 단점

- 화면 코드는 Web과 별도로 작성해야 한다.
- 지도 라이브러리와 백그라운드 위치 추적이 복잡하다.
- 빌드, 권한, 플랫폼 이슈가 추가된다.

### 추천 시점

- 자체 GPS 러닝 기록이 필요할 때
- Android와 iOS를 모두 계획할 때
- 앱 사용 경험을 중요하게 볼 때

---

## 14.4 선택지 D — Kotlin 네이티브 Android

### 장점

- Android 기능을 가장 안정적으로 활용
- 백그라운드 위치 추적
- 센서
- 알림
- Health Connect
- 배터리 관리
- Wear OS 연동

### 단점

- Web과 별도의 코드베이스
- 개발 범위 증가
- iOS는 다시 별도 개발 필요
- 현재 프로젝트의 핵심 기능을 이중 구현할 수 있음

### 추천 시점

- Android만 우선 출시
- 러닝 측정 정확도가 핵심
- Health Connect와 Wear OS가 핵심
- 네이티브 개발 역량 확보

---

## 14.5 권장 Android 방향

현재 iOS 중심 저장소에서는 다음 순서가 가장 합리적이다.

### 1단계

iOS 핵심 경험과 데이터 정확도, 테스트를 먼저 안정화한다.

### 2단계

다른 플랫폼이 필요해지면 공통 데이터 계약과 동기화 API를 먼저 설계한다.

### 3단계

공유·조회 중심이면 Web/PWA, Android 기기 데이터 연동이 우선이면 네이티브 Android를 선택한다.

### 4단계

선택한 플랫폼에서 GPX 업로드 또는 Health Connect 데이터 가져오기를 검토한다.

### 5단계

사용자가 실제로 Run Log 안에서 러닝을 기록하기를 원하는지 검증한다.

### 6단계

Android 수요와 자체 측정 수요가 확인되면 React Native 또는 Kotlin으로 GPS 기록 기능을 개발한다.

# 18. Codex 개발 원칙

Codex는 다음 원칙을 따른다.

1. 먼저 전체 저장소를 읽고 현재 구조를 설명한다.
2. 기존 지도 기능을 다시 만들지 않는다.
3. 기존 프레임워크와 라이브러리를 최대한 유지한다.
4. 한 번에 하나의 기능만 구현한다.
5. 작업 전 변경 대상 파일을 설명한다.
6. 불필요한 대규모 리팩터링을 하지 않는다.
7. 데이터 타입과 상태 흐름을 명확하게 유지한다.
8. 재생 상태의 단일 기준을 사용한다.
9. 재생, 슬라이더, 마커, 지표가 항상 같은 시점을 가리키게 한다.
10. 모바일 환경을 고려한다.
11. timer, listener, animation frame을 정리한다.
12. 기능 구현 후 lint, typecheck, test, build를 실행한다.
13. 실행하지 않은 검증을 성공으로 보고하지 않는다.
14. 미확인 사항은 추측하지 않고 코드에서 확인한다.
15. 변경 범위를 작게 유지한다.
16. 기존 동작을 깨뜨리지 않는다.
17. 기능별 커밋이 가능하도록 작업을 분리한다.
18. 복잡한 기능은 먼저 계획을 작성하고 구현한다.

# 22. 최종 방향

Run Log의 핵심 경쟁력은 다음 순서로 만들어진다.

```text
지도 표시
→ 경로 재생
→ 페이스 시각화
→ 월간 러닝 포트폴리오
→ 과거 기록 비교
→ Ghost Runner
→ 친구 경쟁
→ 게임화
→ 모바일 앱
```

Android 앱을 먼저 만드는 것이 핵심은 아니다.

현재 iOS 앱에서 Run Log만의 차별화된 경험과 데이터 정확도를 먼저 완성한다. Web/PWA는 별도 코드베이스가 아직 없으므로 제품 수요와 공유 요구를 확인한 뒤 착수하고, 이후 사용자에게 자체 러닝 측정 기능과 Android 지원이 정말 필요한지 확인해 네이티브 개발 범위를 결정한다.

초기 제품의 핵심 질문은 다음이다.

> 사용자는 자신이 달린 경로가 지도 위에서 쌓이고, 과거의 자신이나 친구와 비교되는 경험을 반복해서 사용하고 싶어 하는가?

이 질문을 현재 iOS MVP로 먼저 검증한 뒤 Web/PWA와 Android 개발을 확장하는 것이 가장 효율적이다.
